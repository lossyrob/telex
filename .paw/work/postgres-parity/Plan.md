# Plan - Postgres Parity

## Approach Summary

Node outcome anchor: complete Postgres parity for the local presence/transport daemon: the daemon can serve Postgres-backed stores, owns one LISTEN/NOTIFY connection per Postgres store for push wakeups, heartbeats local attendees' backend leases, uses the Postgres backend lease/epoch as the cross-machine source of truth, and proves competing-daemon epoch/fence behavior before the PR can close issue #42.

The implementation will bring the existing Postgres backend up to the daemon-core epoch contract already used by SQLite, wire the daemon to open Postgres store keys from backend profiles, add one daemon-owned Postgres LISTEN/NOTIFY listener per store, and add gated Postgres proof tests for the section 17 axis. Schema/importer hardening is treated as prerequisite work only; the plan ends with backend conformance, daemon-level proof coverage, and local validation evidence.

This is a paw-lite plan: Spec.md, ImplementationPlan.md, and CodeResearch.md are intentionally absent. To offset the reduced artifact coverage, the load-bearing seams that would normally be pinned in CodeResearch.md/ImplementationPlan.md are recorded below as explicit acceptance criteria and key decisions.

## Work Items

- [x] **pg-epoch-backend-parity** - Extend `src/backend/postgres.rs` with epoch-aware schema migration and trait parity: `lease_epoch`, `owner_instance_id`, consumed delivery rows, detach tombstones, backend-clock timestamps, epoch CAS claim, epoch heartbeat/release/reset, `mark_consumed_if_current_owner`, `has_delivery_for_recipient`, retention/pending counts, and message fan-out semantics aligned with SQLite. Acceptance criteria: migration is additive/idempotent; live tables are never dropped/recreated; pre-epoch rows created before `lease_epoch` existed cut over through an explicit `WHERE lease_epoch IS NULL` claim path; old delivered rows are backfilled as consumed with `consumed_at_ms = delivered_at_ms` so already-delivered history is not resurrected; all lease, stale-cutoff, consumed, and tombstone timestamps use the Postgres server clock.
- [x] **daemon-postgres-store-and-notify** - Teach the daemon to resolve `postgres:` store keys back to configured backend profiles, open Postgres stores, start exactly one daemon-owned LISTEN/NOTIFY connection per Postgres store, and use notifications to wake local waiters without adding per-session listeners or weakening poll correctness. Acceptance criteria: Postgres store-key/profile resolution is deterministic and fail-closed on zero or multiple profile matches; no warning-and-first-match behavior; credential resolution uses the same `profiles::pg_connect_config` helper as the CLI at daemon store-open time without logging secrets; the daemon's heartbeat loop uses `heartbeat_epoch` and self-demotes on 0-row/`NotOwner` outcomes; clean drain/shutdown releases locally owned epoch leases; notification setup/health failures are observable and safely degrade to bounded polling without losing durable correctness.
- [x] **postgres-proof-tests** - Add shared/gated Postgres proof coverage for daemon section 17 parity: backend epoch conformance, competing owners/daemon instances over one Postgres schema, `NotOwner` self-demotion, no double-consume/double-deliver across epoch rotation, clean drain/reclaim, and notification multiplexing. Tests must skip cleanly without `TELEX_PG_URL` but fail when `TELEX_PG_REQUIRE=1`. The close-proof checklist is: live Postgres `TELEX_PG_REQUIRE=1` evidence covers named tests for backend epoch CAS, backend-clock heartbeat/reclaim, heartbeat interval vs stale-window safety, predecessor `NotOwner` self-demotion, no double-consume/double-deliver, clean drain successor claim at `epoch+1`, one listener per Postgres store, notify multiplexing to multiple local waiters, and Ack-gated durability. The central Test 6 proof must use independent daemon instances/processes and, for `Closes #42`, containerized multi-host evidence; in-process backend-handle tests are trait coverage only.

Decision matrix: all close-proof items green under live Postgres -> `Closes #42`; live Postgres unavailable -> `Refs #42` with `blocked-on-proof`; any checklist item failing -> `Refs #42` with the failing item named as partial/blocked; preserve the issue's completion condition for orchestrator reconciliation.

## Key Decisions

- Reuse the existing `Backend` epoch APIs rather than adding a Postgres-specific daemon path; daemon behavior should stay backend-polymorphic after `open_store_backend` resolves the store.
- Postgres `deliveries` mirrors SQLite's per-recipient delivery/ack buffer: primary-recipient rows are pending until explicit Ack sets `consumed_at_ms`; CC rows are inserted already consumed; old Postgres rows without `consumed_at_ms` are migrated as consumed at `delivered_at_ms`; `mark_delivered` remains backward-compatibility only and the daemon wait path does not persist consumed state before Ack.
- Keep the Postgres backend clock in the database server clock domain for lease heartbeat/reclaim, stale-cutoff comparisons, consumed timestamps, and detach tombstones. Producer/user-facing timestamps such as `messages.sent_at_ms` stay producer supplied.
- PgBackend v1 uses a simple connection model: one query client per backend instance for normal operations/explicit transactions and one separate daemon-owned LISTEN connection per opened Postgres store. No connection pool is introduced in this node.
- LISTEN/NOTIFY is a wakeup optimization owned by the daemon store entry; durable `fetch_undelivered` + explicit `Ack` remain the correctness path. `pg_notify` is emitted after durable insert; notify failure is surfaced in Status/recent errors and polling fallback bounds wake latency rather than rolling back the durable message.
- The daemon resolves Postgres store keys by loading configured profiles at store-open time, computing each profile's `store_key`, and requiring exactly one profile match. Zero-match and multi-match are actionable fail-closed errors; credentials are resolved through `pg_connect_config` and are never logged.
- Existing pre-epoch Postgres rows are treated as legacy cutover rows after additive migration: `lease_epoch IS NULL` can be claimed through the explicit epoch cutover path, while old delivered rows are marked consumed during migration so they are not resurrected.
- No canonical-store lock is applied to Postgres; the epoch fence is the cross-host/multi-daemon arbiter.
- The close-proof checklist is narrowed to the Postgres axis of daemon.md section 17 test 6 plus required clean drain/reclaim proof for this node. Broader ordered-handoff crash-matrix coverage remains owned by `fencing-proof`/downstream validation unless the issue is explicitly amended.
